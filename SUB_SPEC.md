Понимаю проблему. Это классическая ловушка при создании авто-мапперов: сырая математика (даже производные) слишком «жадная». Она видит акустический удар там, где человеческий мозг его фильтрует (микро-отскоки палочек, тихие хэты, резонансы). Для ритм-игры важен не *каждый* звук, а **грув (groove) и музыкальная абстракция**. Игрок хочет отбивать пульс трека, а не стенографировать шум.

Чтобы сделать карту играбельной, нам нужно перевернуть подход. Вместо того чтобы искать все пики и пытаться притянуть их к сетке, мы **сначала найдем сетку (BPM), а затем будем проверять, есть ли акцентированный звук на узлах этой сетки**.

Вот полностью переработанный `SPEC.md`, ориентированный на музыкальность, жесткий контроль плотности (anti-spam) и строгую асинхронность для плавности UI.

# SPEC.md — Подсистема генерации Beatmap (Grid-Scoring & Density Control)

## 1. Философия и Архитектура

Новый алгоритм отказывается от слепого "Peak Picking". Пайплайн строится на принципе **Grid-First (Сначала сетка)**. Мы вычисляем пульсацию трека, строим идеальную музыкальную сетку (четверти, восьмые доли) и "опрашиваем" аудиосигнал только в этих контрольных точках.

**Главные изменения:**

1. **Музыкальная абстракция:** Игнорируем всё, что не попадает в сильные и полусильные доли, если только это не ярко выраженная ритмическая сбивка (fill).
2. **Ограничитель плотности (Stamina/Density Control):** Алгоритм понимает, насколько плотно стоят ноты, и динамически завышает порог вхождения для новых нот, чтобы не создавать "непроходимые" стены (spam).
3. **100% Асинхронность:** Все тяжелые циклы разбиты на чанки через `Event Loop`, чтобы анимация загрузки (CSS) в UI не зависала ни на миллисекунду.

## 3. Фазы разработки алгоритма (Атомарная декомпозиция)

### Фаза 1: Асинхронное декодирование (Decoding & Downmix)

**Цель:** Получить моно-сигнал, не блокируя UI.

1. **`await AudioContext.decodeAudioData()`** — нативно асинхронная операция.
2. **Асинхронный Downmix:** Сложение левого и правого каналов:
$PCM[i] = (L[i] + R[i]) \times 0.5$
*Требование:* Если сэмплов миллионы, разбиваем цикл `for` на блоки по 1,000,000 итераций. Между блоками вызываем `await new Promise(r => setTimeout(r, 0))` для передачи управления рендеру браузера.

### Фаза 2: Multi-Band Isolation (Многополосная фильтрация)

**Цель:** Разделить трек на 3 смысловых слоя: Kick (Бочка), Snare (Рабочий), Hi-Hat (Хэты/Перкуссия).

1. Создать `OfflineAudioContext`.
2. Настроить 3 параллельные цепи `BiquadFilterNode`:
* **Low (Кик):** Low-pass, 120 Hz.
* **Mid (Снейр):** Band-pass, 1000-3000 Hz.
* **High (Хэты):** High-pass, 6000 Hz.


3. Отрендерить 3 независимых `AudioBuffer` (`await offlineCtx.startRendering()`).
4. **Асинхронное получение огибающих (Envelopes):** Пройти по каждому из 3-х буферов, применить модуль (Rectification) и легкий Low-pass (сглаживание). На выходе получаем три массива огибающих: $E_{low}, E_{mid}, E_{high}$.

### Фаза 3: Global Tempo Detection (Поиск пульса)

**Цель:** Найти точный BPM до расстановки нот.

1. Взять только $E_{low}$ (бочку, так как она задает фундамент трека).
2. Вычислить первую производную, чтобы найти точки резких атак (Onsets).
3. Вычислить расстояния (дельты) между сильными атаками.
4. Построить взвешенную гистограмму интервалов и определить базовый BPM.
5. **Поиск Offset (Сдвига):** Найти первый мощный удар бочки в начале трека — это будет наша точка отсчета (Доля 1.1).

### Фаза 4: Grid-Based Energy Scoring (Оценка узлов сетки)

**Цель:** Проверить аудио только там, где логически могут быть ноты.

1. **Генерация сетки:** Исходя из BPM и Offset, генерируем временные метки для 1/4 (сильные доли), 1/8 (слабые доли) и 1/16 (быстрые сбивки) долей на весь трек.
2. **Scoring (Оценка):** Для каждого узла сетки $t$ смотрим "окно" в аудиоданных (например, $\pm 20$ мс) во всех трех полосах.
3. Вычисляем общий балл удара:
$Score(t) = (W_{low} \cdot E_{low}) + (W_{mid} \cdot E_{mid}) + (W_{high} \cdot E_{high})$
Где веса $W$ расставлены в пользу сильных ударов (например, Кик = 1.0, Снейр = 0.8, Хэт = 0.3).

### Фаза 5: Intelligent Culling (Ограничитель спама / Density Control)

**Цель:** Превратить сырые баллы в играбельный ритм, отсекая лишнее.

1. Проходим по узлам сетки в хронологическом порядке.
2. **Динамический порог (Dynamic Threshold):** * Для 1/4 долей порог прохождения $Score(t)$ минимален (разрешаем почти всё).
* Для 1/8 долей порог выше (ставим ноту, только если звук реально отчетливый).
* Для 1/16 долей порог максимален (ставим, только если это мощная барабанная сбивка).


3. **Stamina Meter (Шкала усталости):**
* Вводим переменную `densityPenalty`.
* Каждый раз, когда мы ставим ноту 1/8 или 1/16, `densityPenalty` возрастает.
* Этот штраф вычитается из $Score(t)$ следующих узлов.
* Со временем (на длинных нотах 1/4 или пустотах) штраф остывает (уменьшается).
* *Результат:* Алгоритм физически не сможет поставить стену из 30 нот подряд — после 5-6 быстрых ударов он "устанет" и пропустит слабые звуки, оставляя только акценты, создавая естественные паузы для рук.



### Фаза 6: Pattern Generation (Раскладка D F J K)

**Цель:** Красиво распределить ноты по 4 линиям.

1. Сетки разделены на такты (по 4 удара).
2. **Логика приоритетов:**
* Если узел преодолел порог только за счет $E_{low}$ (Бочка) — ставим ноту на левую руку (D или F).
* Если за счет $E_{mid}$ (Снейр) — на правую руку (J или K).
* Если $Score(t)$ превышает экстремальный порог (Crash/Удар в тарелку + Бочка) — ставим **Аккорд** (например, D + K).


3. **Rhythm Flow:** Запрещаем паттерны типа D-D-D-D на 1/8 долях (Jack-spam). Принудительно разворачиваем их в триллы (D-F-D-F) или лестницы.

### Фаза 7: Асинхронная обёртка (API & UI Thread)

**Цель:** Строгая интеграция с `app.js` и `publishForm.js` без "фризов".

1. Весь алгоритм оборачивается в класс `BeatmapBuilder` с главным методом `async generate(audioBuffer, difficultyMultiplier)`.
2. **Chunking-функция:**
```javascript
// Утилита для освобождения Event Loop
const yieldToMain = () => new Promise(resolve => setTimeout(resolve, 0));

```


3. Во всех циклах Фаз 3, 4 и 5 (где итерации идут по миллионам сэмплов или десяткам тысяч узлов сетки) внедрить счетчик:
```javascript
for (let i = 0; i < data.length; i++) {
    // ... вычисления ...
    if (i % 50000 === 0) await yieldToMain();
}

```


4. Функция должна принимать коллбэк `onProgress(percent)` для обновления красивого прогресс-бара из `prototype/code.html`.

## 4. Почему это решит проблему?

1. **Нет шума:** Мы проверяем звук *только* в музыкально обоснованных точках времени (узлах сетки). Если в треке есть случайный щелчок вне ритма — алгоритм туда даже не посмотрит.
2. **Дыхание ритма:** `Stamina Meter` (Фаза 5) работает как виртуальный игрок. Он не позволит создать мапу, от которой отвалятся пальцы, заставляя алгоритм выделять только самые сочные удары, если плотность слишком высока.
3. **Разделение инструментов:** Мы больше не путаем тихий бас с громким хэтом (Фаза 2 и 4), так как каждая частотная полоса имеет свой музыкальный "вес".
